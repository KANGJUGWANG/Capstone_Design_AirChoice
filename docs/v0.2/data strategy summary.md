# v0.2 데이터 수집 및 API 전략 정리

## 1. 문서 목적
본 문서는 프로젝트 초기 단계에서 진행한 API 탐색, 실패 원인, 전략 변경 과정과 향후 데이터 수집 및 설계 방향을 정리한다.

## 2. 초기 목표
- 항공권 가격 데이터를 API 기반으로 수집
- 실시간 또는 준실시간 가격 기반 모델 구축
- 의사결정 보조(구매 vs 관망)

## 3. 초기 API 탐색 및 실패 사례

### 3.1 Amadeus
- 상태: 사용 불가
- 확인 내용:
  - 가입/접근 제약으로 즉시 사용 불가
  - 초기 기준 API 후보였으나 실제 진행 불가
- 판단:
  - 현 단계 메인 공급자로 채택 불가

### 3.2 Skyscanner 공식 API
- 상태: 즉시 사용 불가
- 확인 내용:
  - 파트너 승인형 접근 구조
  - 셀프서비스 즉시 실험 불가
- 판단:
  - 장기 검토 대상
  - 초기 데이터 수집 메인 소스로 부적합

### 3.3 Kiwi API
- 상태: 보류
- 확인 내용:
  - 접근성/문서 확인이 불안정함
  - 안정적으로 바로 붙일 수 있는 구조 확인 실패
- 판단:
  - 현재 기준 우선순위 낮음

## 4. 대안 탐색 및 현재 후보

### 4.1 Travelpayouts
- 상태: 사용 가능
- 확인 내용:
  - 토큰 확보 완료
  - 실제 응답 확인 성공
  - 가격, 항공사, 출발/귀국 시각, 만료시각, 소요시간 수집 가능
- 특징:
  - 캐시 기반 관측 데이터
  - 사용자 검색 기반 가격
  - `expires_at` 존재
- 현재 역할:
  - 메인 누적 데이터 후보

### 4.2 SerpApi
- 상태: 검토 대상
- 확인 내용:
  - Google Flights 검색 결과 기반 구조화 API
  - 현재 검색 결과 관측에 적합
- 특징:
  - 실시간/준실시간 비교 검증용으로 적합
  - 비용 및 호출량 제약 고려 필요
- 현재 역할:
  - 시연/검증용 후보

### 4.3 RapidAPI
- 상태: 채널 수준 후보
- 확인 내용:
  - 공급자별 API 품질 편차 큼
  - 단일 API로 취급할 수 없음
- 특징:
  - API별 안정성, 비용, 응답 구조 별도 검토 필요
- 현재 역할:
  - 개별 공급자 후보 탐색 채널

## 5. 핵심 문제 인식

### 5.1 데이터 유형 차이
- Kaggle: 과거 관측 데이터
- Travelpayouts: 캐시 기반 관측 데이터
- SerpApi: 현재 검색 기반 관측 데이터

정리:
- 모두 결제 장부 데이터는 아님
- 모두 "관측 가격 데이터"로 해석 가능
- 단, 공급자별 생성 방식과 분포는 다름

### 5.2 데이터 드리프트
- Kaggle ↔ API: 분포 차이 큼
- Travelpayouts ↔ SerpApi: 분포 차이 존재

정리:
- `provider` 구분 필요
- 모든 공급자를 동일 분포로 보면 안 됨

### 5.3 가격 정의 문제
- 좌석 재고 기반 실제 결제 가격 아님
- 검색 시점/캐시 시점 가격
- 공급자별 가격 산출 맥락 다름

정리:
- 절대 가격 예측보다
- 분포 기반 구매/관망 판단이 적합

## 6. 전략 수정

### 변경 전
- 단일 API 기반 가격 예측

### 변경 후
- 관측 가격 데이터 기반 분포 학습
- 구매/관망 2진 분류 문제로 접근
- 메인 누적 소스 + 보조 검증 소스 구조

## 7. Travelpayouts 실응답 확인 결과

### 확인된 쿼리
- `ICN -> TYO`
- `ICN -> NRT`
- `ICN -> HND`

### 확인 결과
- `ICN -> TYO`: 응답 존재
- `ICN -> NRT`: 응답 존재, 결과 목적지 `TYO`로 반환
- `ICN -> HND`: 응답 존재, 결과 목적지 `TYO`로 반환
- 일부 쿼리: `data = {}` 빈 응답 존재

### 확인된 의미
- 공항 단위 요청 가능
- 결과는 도시 단위로 묶여 반환될 수 있음
- 저장 시 `query_destination`과 `result_destination` 분리 필요

## 9. Provider 역할 정의(현재안)
- Travelpayouts: 메인 누적 데이터
- SerpApi: 검증 및 시연 비교
- Kaggle: 베이스라인 학습 보조

## 10. MVP 범위(현재안)
- 기본 분석 단위: `ICN ↔ TYO`
- 요청 코드: `TYO`, `NRT`, `HND` 병행 검토

## 11. 현재까지의 핵심 결론
- 공식 API 중심 초기 계획은 현실적으로 무너짐
- 실제 붙은 가격 관측 소스는 현재 Travelpayouts
- Travelpayouts를 바로 확정하기보다, SerpApi/RapidAPI 계열과 비교 평가가 필요
- 지금 우선순위는 모델보다 데이터 수집 구조와 스키마 검증

## 12. 다음 단계

3. 그 결과를 기준으로 Kaggle 데이터셋 선정

## 13. 핵심 정리
- 모든 데이터는 "관측 가격 데이터"다
- 절대 가격 예측보다 구매/관망 의사결정 보조가 적합하다
- 공급자는 단일 확정이 아니라 역할 분리 구조로 검토해야 한다
