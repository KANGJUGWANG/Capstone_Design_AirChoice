---

## 1. 문서 목적

본 문서는 v0.4.3 기준 MVP 앱 구조, 화면 구성, 기술 스택, 서비스 방식에 대한 현재 확정 사항을 정리한다.

## 2. 상태 구분 기준

- **확정**: 현재 기준으로 채택한 항목
- **보류**: 후보로 유지하되 현재 메인 진행 대상이 아닌 항목

## 4. 기술 스택

| 레이어 | 기술 | 상태 |
|---|---|---|
| 프론트엔드 | React (웹앱) | 확정 |
| 백엔드 | FastAPI | 확정 |
| DB | MySQL 8.0.45 (기존 Oracle Cloud) | 확정 |
| 인증 | 카카오 로그인 | 확정 |
| 배포 | Vercel 또는 Netlify | 미확정 |
| 앱 서버 위치 | 학교 서버 스펙 확인 후 결정 | 미확정 |

### 프론트엔드 폴더 구조 원칙

React Native 전환 가능성을 고려하여 아래 구조로 분리한다.

```
src/
  api/        ← API 호출 (React Native 재사용 가능)
  hooks/      ← 커스텀 훅 (재사용 가능)
  store/      ← 상태 관리 (재사용 가능)
  components/ ← UI 컴포넌트 (웹 전용)
  screens/    ← 화면 단위 (웹 전용)
```

## 8. 화면별 기준

### 8-1. 홈

- 검색 입력만 존재, 미니멀
- 한 화면에 모두 들어오도록 구성
- 출발지 / 도착지 / 날짜 / 편도·왕복 선택

### 8-2. 검색 결과

- 항공권 카드 리스트 표출
- 각 카드에 모델 분류 태그 부착 (모델 미완성 시 빈 상태 처리)
- 우상단에 검색 상세보기 진입 버튼 (작게, 카드 화면 방해 없이)
- 카드 탭 → 카드 상세보기 진입

### 8-3. 카드 상세보기

- 해당 항공권 BUY/WAIT 의사결정 보조
- 모델 출력 방식 확정 전까지 세부 구성 보류
- 모델 데이터 자리 유지, 데이터 없음 처리
- 카드 저장 버튼 포함

### 8-4. 검색 상세보기

- 해당 검색 노선 전체 기준 추가 정보
- 요일 패턴 표시 (데이터 확보 전까지 자리만 유지)
- 모델 관련 추가 정보
- 검색 결과 화면과 관점 분리: "이 항공권" vs "이 노선 전체"

### 8-5. 저장 목록

- 알람 개념 아닌 저장 개념
- 저장 시점부터 현재까지의 가격 변동 표시
- 저장 시점 판단 + 현재 판단 비교 표시

### 8-6. 사용자 설정

- 계정 정보 (카카오 연동 상태)
- 알림 설정

### 8-7. 모델 정보

- 사용 모델 설명
- 모델 완성 후 내용 채움
- MVP에서는 자리만 존재

### 8-8. 회원가입 / 로그인

- 카카오 로그인 버튼
- 가볍게 구성
