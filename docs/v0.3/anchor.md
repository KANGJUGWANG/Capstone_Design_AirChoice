# v0.3 Project Anchor

---

## 1. 문서 목적

본 문서는 v0.2.1 이후 진행된 API 탐색 실패, 방향 전환, 신규 API 검증 결과를 정리하여 현재 프로젝트의 기준점(Anchor)으로 사용한다.

다음 목적을 가진다.

- v0.2.1 방향이 폐기된 원인과 판단 근거 명시
- v0.3 기준으로 확정된 항목과 미확정 항목 구분
- 이후 진행 시 판단 기준이 흔들리지 않도록 기준점 역할 수행

---

## 2. 상태 구분 기준

- **확정**: 실호출 또는 검증을 통해 채택한 항목
- **보류**: 후보로 유지하되 현재 메인 진행 대상이 아닌 항목
- **미확정**: 후속 검증 후 결정할 항목
- **폐기**: 이전 버전에서 검토했으나 현재 진행 대상에서 제외한 항목

---

## 3. v0.2.1 방향 폐기 사유

v0.2.1은 Travelpayouts Data API를 메인 수집 소스로 채택한 방향이었다. 그러나 실제 진행 과정에서 다음 문제가 확인되었다.

- Travelpayouts는 Aviasales 사용자 검색 이력 기반 캐시 데이터 구조임
- 캐시 특성상 `success=True`여도 `data={}` 빈 응답이 빈번하게 발생하며 수집 안정성이 낮음
- 수집 요청 목적지와 실제 응답 목적지가 다르게 반환되는 구조적 문제 존재
- 누적 수집 기반 DPD 시계열 구성에 적합하지 않다는 판단

위 이유로 v0.2.1에서 설계한 전체 수집 구조를 폐기하고, API 탐색을 신규로 재시작했다.

## 5. 현재 확정 사항

### 5-1. API

| 항목 | 내용 | 상태 |
|---|---|---|
| API 공급자 | FlightAPI.io | 확정 |
| 편도 엔드포인트 | `onewaytrip` | 확정 |
| 왕복 엔드포인트 | `roundtrip` | 확정 |
| 과금 구조 | 요청당 2크레딧 (Standard 플랜 기준) | 확정 |
| ICN → NRT 데이터 반환 | 실호출 확인 완료 | 확정 |
| ICN → HND 데이터 반환 | 실호출 확인 완료 | 확정 |
| 가격 통화 | KRW 반환 확인 | 확정 |

### 5-2. Places 확정

| place_id | display_code | name | 비고 |
|---|---|---|---|
| 12409 | ICN | Incheon International | 출발 공항 |
| 14788 | NRT | Tokyo Narita | 도착 공항 (NRT 요청 시) |
| 12234 | HND | Tokyo Haneda | 도착 공항 (HND 요청 시) |
| 7968 | TYO | Tokyo | 도시 상위 노드 |

NRT와 HND는 동일 parent_id(7968, Tokyo)에 속하며 별도 요청으로만 구분된다.

### 5-3. 수집 범위

| 항목 | 내용 | 상태 |
|---|---|---|
| 대상 노선 | ICN → NRT, ICN → HND (각각 별도 요청) | 확정 |
| 왕복 노선 | NRT → ICN, HND → ICN (각각 별도 요청) | 확정 |
| DPD 상한 | 120일 | 확정 |
| 왕복 체류 기간 | 7일 고정 | 확정 |
| 수집 주기 | 매일 반복 수집 | 확정 |

### 5-4. 모델 방향

| 항목 | 내용 | 상태 |
|---|---|---|
| 문제 유형 | BUY or WAIT 이진 분류 | 확정 |
| 핵심 feature | DPD, 현재가격, 항공사, 출발 요일, 직항 여부 | 확정 |
| 가격 해석 기준 | 관측 가격 (결제 장부 아님) | 확정 |

### 5-5. Kaggle 보조 데이터

| 항목 | 내용 | 상태 |
|---|---|---|
| 후보 데이터셋 | dilwong/FlightPrices | 확정 |
| 구조 적합성 | searchDate + flightDate → DPD 직접 계산 가능 | 확정 |
| DPD 커버리지 | DPD 1~65만 존재. 66~120은 직접 수집으로만 커버 | 확정 |
| 최종 활용 여부 | 누적 30일 후 직접 수집 데이터량 보고 결정 | 미확정 |
