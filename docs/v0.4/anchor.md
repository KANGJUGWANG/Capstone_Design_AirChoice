# v0.4 Project Anchor

---

## 1. 문서 목적

본 문서는 v0.3 이후 진행된 수집 방식 전환, 저장 구조 구체화, 결과물 방향 확장을 반영하여 현재 프로젝트의 기준점(Anchor)으로 사용한다.

다음 목적을 가진다.

- v0.3 기준에서 유지되는 항목과 변경되는 항목 구분
- API 기반 수집 방향 종료 및 크롤링 기반 수집 방향 전환 사유 명시
- v0.4 기준으로 확정 항목, 미확정 항목, 폐기 항목 재정리
- 이후 문서(`baseline`, `research`, `schema`, `data_pipeline`, `checklist`, `product`)의 상위 기준점 역할 수행

---

## 2. 상태 구분 기준

- **확정**: 현재 운영 기준으로 채택한 항목
- **보류**: 후보로 유지하되 현재 메인 진행 대상이 아닌 항목
- **미확정**: 후속 검증 또는 구현 결과 확인 후 결정할 항목
- **폐기**: 이전 버전에서 검토했으나 현재 진행 대상에서 제외한 항목

## 4. v0.3 대비 주요 변경 사항

v0.4는 단순 진행 업데이트가 아니라, 수집 방식과 운영 구조가 변경된 버전이다.

| 구분 | v0.3 기준 | v0.4 기준 |
|---|---|---|
| 수집 소스 | FlightAPI.io | Google Flights |
| 수집 방식 | API 호출 | 브라우저 자동화 + 인터셉트 기반 수집 |
| 응답 구조 | 구조화된 API 응답 | 인터셉트 원본 + 파싱 결과 |
| 저장 구조 | API 응답 기반 스키마 중심 | raw 파일 저장 + 파싱 + DB 적재 구조 |
| 운영 비용 가정 | 과금형 API 포함 | 무료 인프라 우선 (Oracle Cloud A1) |
| 결과물 방향 | 수집/모델 중심 | 분석 대시보드형 앱까지 포함 |

---

## 5. 방향 전환 사유

v0.3에서는 FlightAPI.io를 메인 수집 API로 채택했으나, 이후 진행 과정에서 아래 사유로 방향을 전환했다.

- 캡스톤 운영 관점에서 비용 발생 API 지속 사용이 부담 요인으로 작용
- 지도 교수 권유에 따라 유료 API보다 직접 수집 구현 경험을 우선 고려
- 서비스형 결과물과 연결 시, 페이지 기반 실제 검색 결과 수집 구조 검토 필요
- Google Flights를 기준 페이지로 두고 직접 수집하는 방향이 프로젝트 실현성과 산출물 연결 측면에서 더 적합하다고 판단

따라서 v0.4에서는 FlightAPI.io 기반 수집 구조를 메인 경로에서 제외하고, Google Flights 인터셉트 기반 수집 구조를 기준 방향으로 재설정한다.

## 6. 현재 확정 사항

### 6-1. 수집 대상 및 방식

| 항목 | 내용 | 상태 |
|---|---|---|
| 기준 대상 페이지 | Google Flights | 확정 |
| 수집 방식 | HTML 직접 파싱이 아닌 인터셉트 방식 우선 검토 | 확정 |
| 수집 주기 | 1일 3회 (8시간 간격) | 확정 |
| 여행 유형 | 편도 + 왕복 동시 수집 | 확정 |
| DPD 상한 | 120일 | 확정 |
| 대상 축 | 인천 ↔ 도쿄 기준 노선 조합 수집 | 확정 |

### 6-2. 인프라 및 저장 방향

| 항목 | 내용 | 상태 |
|---|---|---|
| 수집 서버 | Oracle Cloud A1 Free Tier | 확정 |
| 데이터 저장소 | MySQL 우선 사용 | 확정 |
| 보조 저장 형태 | CSV / JSON fallback 가능 | 확정 |
| raw 저장 원칙 | 원본 인터셉트 데이터 보존 | 확정 |
| 파이프라인 방향 | 수집 → raw 저장 → 파싱 → 정규화 → 적재 | 확정 |

### 6-3. 결과물 방향

| 항목 | 내용 | 상태 |
|---|---|---|
| 결과물 형태 | 분석 대시보드형 앱 | 확정 |
| 와이어프레임 상태 | Manus 기반 프로토타입 초안 생성 완료 | 확정 |
| 참조 자산 | `docs/v0.4/assets/` 내 GIF 5종 기준 | 확정 |
