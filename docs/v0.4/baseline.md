# v0.4 Baseline — 주제 흐름 및 설계 기준

---

## 1. 문서 목적

본 문서는 v0.4 기준 프로젝트의 문제 정의, 핵심 해석 축, 수집 구조 해석, 결과물 방향을 정리한다.

설계 판단이 흔들릴 경우 본 문서를 기준으로 재확인한다.

---

## 2. 프로젝트 핵심 방향

### 2-1. 문제 정의

항공권 가격은 동일 노선·유사 조건이더라도 관측 시점에 따라 다르게 형성된다.  
소비자는 현재 가격이 충분히 낮은 상태인지, 더 기다릴 가치가 남아 있는지에 대한 정량적 기준 없이 구매 결정을 내리는 경우가 많다.

본 프로젝트는 이 문제를 **항공권 구매 의사결정 보조**의 형태로 다룬다.

### 2-2. 해석 기준

본 프로젝트의 목적은 **전역 최적 구매일 탐색**이 아니다.

현재 시점에서 아래 질문에 답하는 것을 목표로 한다.

- 지금 사는 것이 유리한가
- 더 기다릴 가치가 남아 있는가
- 기다릴 경우 가격 상승 위험이 큰가
- 현재 가격은 최근 관측 흐름 대비 어떤 위치에 있는가

즉, 본 프로젝트는  
**“최적 구매일 예측”보다 “현재 시점의 WAIT 가치평가(확률적 의사결정 보조)”**를 목표로 한다.

### 2-3. 문제 유형

- 문제 유형: **BUY or WAIT 이진 분류**
- 핵심 해석 축: **DPD (Days to Departure, 출발까지 남은 일수)**
- 판단 방식: 현재 가격과 이후 가격 변화의 비교를 통한 사후적 라벨 생성
- 출력 형태: BUY / WAIT 권고, 점수화, 요인 분석

## 3. 결과물 방향

본 프로젝트의 결과물은 단순 모델 성능 보고서에 한정하지 않는다.

최종 결과물은 아래 두 축을 함께 포함한다.

| 축 | 내용 |
|---|---|
| 데이터/모델 축 | 항공권 관측 데이터 수집, 가격 흐름 분석, BUY/WAIT 판단 로직 |
| 서비스/앱 축 | 가격 요약, BUY/WAIT 표시, 요인 분석, 알림 기능을 포함한 대시보드형 앱 |

v0.4에서는 데이터 수집 구조와 함께 **분석 대시보드형 결과물**을 병행 설계 대상으로 둔다.

---

## 4. 기준 데이터 소스와 수집 방식

### 4-1. 기준 소스

v0.4 기준 수집 대상은 **Google Flights**이다.

### 4-2. 수집 방식

수집 방식은 브라우저 자동화 기반의 **인터셉트 방식**을 우선 검토한다.

HTML 본문을 직접 파싱하는 방식은 아래 한계가 있다.

- 수집 범위 대비 처리 비용이 큼
- 편도 + 왕복 + DPD 120 + 다중 노선 조합에 비효율적
- 화면 구조 변경 시 유지보수 부담이 큼

따라서 v0.4에서는 **HTML 직접 파싱을 메인 경로로 두지 않고**,  
인터셉트 응답을 원본으로 확보한 뒤 필요한 필드를 파싱하는 방향을 기준으로 한다.

### 4-3. 변경 의미

이 변경은 단순 구현 방식 변경이 아니라,  
v0.3의 API 기반 수집 설계에서 **직접 수집 기반 설계로 기준을 전환한 것**을 의미한다.

---

## 5. 핵심 변수와 해석 기준

### 5-1. 핵심 변수

현재 기준 핵심 변수는 아래와 같다.

- `observed_at` : 관측 시점
- `departure_date` : 출발일
- `dpd` : 출발까지 남은 일수
- `price` : 관측 가격
- `carrier_tag` : 항공사 태그 또는 항공사 식별 정보
- `route` : 노선 정보
- `trip_type` : 편도 / 왕복
- `stop_info` : 직항 / 경유 정보
- `departure_weekday` : 출발 요일

### 5-2. 가격 해석

본 프로젝트에서 사용하는 가격은 **실제 결제 장부 데이터**가 아니라,  
관측 시점의 검색 결과 기반 **관측 가격**이다.

따라서 분석 대상은 “실매출 가격”이 아니라,  
**소비자가 현재 검색 시점에 실제로 마주하는 가격 정보**로 해석한다.

## 6. 데이터 구조 해석

### 6-1. 수집 구조 개요

매일 반복 수집을 수행하면 데이터는 아래 구조로 누적된다.

- X축: 출발 예정일
- Y축: DPD
- 각 셀: 특정 출발일을 특정 DPD 시점에서 관측한 가격

이 구조는 직사각형이 아니라 **평행사변형(또는 마름모형 누적 구조)**에 가깝다.

### 6-2. 구조적 특징

수집 기간이 30일이라고 가정할 때,  
DPD 120에서 시작한 출발일은 수집 종료 시점에 DPD 90 부근까지 관측된다.  
즉 모든 출발일에 대해 전체 DPD 구간이 완전히 채워지지는 않는다.

### 6-3. 활용 가능 범위

이 구조는 아래 목적에는 활용 가능하다.

- DPD 기준의 로컬 가격 패턴 해석
- 현재 시점 BUY/WAIT 판단
- 특정 DPD 시점의 기다림 가치 평가
- 동일 조건군 내 단기 가격 흐름 비교

### 6-4. 제한 사항

이 구조만으로 아래를 강하게 주장하는 것은 제한적이다.

- 연간 계절성 일반화
- 출발월 효과 분리
- 전체 가격 곡선 복원
- 모든 노선에 공통인 절대 최적 구매일 도출

따라서 본 프로젝트는  
**전체 시즌성 분석보다 DPD 기준의 로컬 의사결정 보조 문제로 해석**한다.

---

## 7. 수집 범위 기준

### 7-1. DPD 상한

**DPD 상한은 120일**로 둔다.

이 값은 아래 균형을 기준으로 설정한다.

- 실제 구매 의사결정이 발생하는 유효 구간 포함
- 과도하게 먼 미래 구간 제외
- 수집 비용 및 운영 부담 통제

### 7-2. 왕복 체류 기간

왕복은 **체류 기간 7일 고정**을 기준으로 둔다.

이는 편의적 선택이 아니라,  
체류 기간 효과를 통제하여 DPD 효과를 분리하기 위한 분석 조건이다.

### 7-3. 수집 주기

기준 수집 주기는 **1일 3회 (8시간 간격)** 이다.

이는 급격한 단기 변동 구간을 놓치지 않으면서도,  
무료 서버 환경과 운영 부담을 함께 고려한 절충안이다.

---

## 8. 서비스 구조와 화면 방향

현재 결과물은 분석 결과를 사용자에게 직접 보여주는 **대시보드형 앱**을 전제로 한다.

현재 구상된 핵심 화면 축은 아래와 같다.

| 화면 | 기능 |
|---|---|
| 검색 & 요약 | 노선 입력, 현재 가격 요약, 급락 노선 표시 |
| 가격 분석 | BUY/WAIT 점수 및 현재 권고 |
| 요인 분석 | 현재 판단에 영향을 준 요인 설명 |
| 알림 설정 | 목표가 / 골든타임 알림 |
| 환경 설정 | 운영 기준 및 사용자 설정 |

와이어프레임/프로토타입은 별도 자산으로 관리하며,  
v0.4에서는 이 방향성을 공식 결과물 구조로 인정한다.

---

## 9. 외부 데이터 활용 기준

### 9-1. 보조 데이터 역할

외부 공개 데이터(Kaggle 등)는 직접 수집 데이터의 대체재가 아니다.

역할은 아래로 제한한다.

- 사전학습 보조
- 패턴 형태 학습 보조
- 파이프라인 검증 보조

### 9-2. 활용 제한

외부 데이터는 아래 항목에 직접 사용하지 않는다.

- 최종 라벨 기준 결정
- 최종 BUY/WAIT 임계값 결정
- 절대 가격 수준 이전
- 최종 확률 calibration

최종 판단 기준은 반드시 직접 수집 데이터 기준으로 설정한다.

## 10. 현재 기준 해석

v0.4 기준 프로젝트는 아래와 같이 요약한다.

> Google Flights 기반 관측 데이터를 반복 수집하고,  
> DPD 기준의 단기 가격 흐름을 해석하여  
> 현재 시점에서의 BUY/WAIT 판단과 기다림 가치평가를 제공하는  
> 분석 대시보드형 캡스톤 프로젝트
