## 2. v0.3까지의 탐색 결과 요약

v0.3까지는 **유료 API 기반 수집**을 중심으로 탐색했다.

확인된 주요 결과는 아래와 같다.

| 구분 | 결과 | 적용 판단 |
|---|---|---|
| Travelpayouts | 캐시 기반, 빈 응답/목적지 불일치 발생 | 폐기 |
| FlightAPI.io | 실호출 가능, 구조화 응답 확인 | v0.3 기준 채택 |
| SerpApi 계열 | 후보성은 있으나 비용/공급 구조 부담 존재 | 보류 |
| 제휴/승인형 API (예: KAYAK, 일부 공식 API) | 승인형 구조, 즉시 사용 어려움 | 제외 |

v0.3 기준에서는 FlightAPI.io를 메인 수집 경로로 채택했으나, 이후 v0.4에서 **비용 발생 API 의존을 줄이고 직접 수집 구현으로 방향을 전환**했다.

## 3. API 기반 수집 방향 종료 사유

### 3-1. 비용 지속성 문제

API 기반 수집은 구조적으로 구현이 단순하고 응답 파싱이 쉬운 장점이 있으나,  
캡스톤 운영 기간 동안 반복 수집을 지속할 경우 비용이 누적된다.

본 프로젝트는 아래 조건을 전제로 수집을 계획했다.

- 편도 + 왕복
- DPD 120
- 인천 ↔ 도쿄 기준 다중 노선 조합
- 1일 3회 반복 수집

이 조건에서는 과금형 API를 장기간 메인 경로로 유지하는 것이 부담 요인으로 작용한다.

### 3-2. 캡스톤 성격상 직접 구현 경험의 가치

본 프로젝트는 단순 모델 실험보다 **데이터 확보 → 저장 → 파싱 → 분석 → 서비스 연결**까지 포함하는 캡스톤 결과물을 목표로 한다.

따라서 직접 수집 구조를 구현하고 운영하는 경험이 프로젝트 목적과 더 잘 맞는다고 판단했다.

### 3-3. 서비스형 결과물과의 결합성

최종 산출물이 분석 대시보드형 앱을 포함하는 방향으로 구체화되면서,  
실제 검색 화면에서 어떤 데이터를 확보할 수 있는지와 그 흐름 자체가 결과물의 일부가 되었다.

이 점에서 페이지 기반 직접 수집 구조는 서비스 결과물과 연결성이 높다.

## 4. 크롤링 방향 전환

### 4-1. 전환 배경

v0.4에서는 지도 교수 피드백과 운영 현실성을 반영하여,  
유료 API 기반 수집 대신 **크롤링 기반 수집**을 기준 방향으로 전환했다.

### 4-2. 전환 의미

이 전환은 단순 구현 방식 변경이 아니다.

아래 항목이 함께 변경된다.

| 항목 | v0.3 | v0.4 |
|---|---|---|
| 수집 소스 | API 응답 | 페이지 기반 검색 결과 |
| 수집 방식 | 정형 응답 호출 | 브라우저 자동화 + 인터셉트 |
| 저장 단위 | API 응답 중심 | raw 인터셉트 + 파싱 결과 |
| 운영 가정 | 과금형 API 포함 | 무료 인프라 우선 |
| 설계 중심 | 응답 필드 매핑 | 수집/저장/파싱/적재 파이프라인 |

따라서 v0.4는 “수집 방식 세부 수정”이 아니라,  
**수집 구조 자체를 재설계한 버전**으로 해석한다.

## 5. 기준 대상 페이지 선정 — Google Flights

### 5-1. 선정 이유

v0.4 기준 수집 대상은 **Google Flights**로 둔다.

선정 이유는 아래와 같다.

| 기준 | 내용 |
|---|---|
| 사용자 관점 | 소비자가 실제로 가격 비교를 수행하는 대표적 검색 환경 중 하나 |
| 서비스 적합성 | “지금 살까? 기다릴까?” 형태의 의사결정 보조와 연결이 쉬움 |
| 결과물 연결성 | 검색/요약/BUY-WAIT/요인 분석 화면과 직접 연결 가능 |
| 데이터 활용성 | 항공사 태그 기반 가격 식별 가능성 검토 가치 존재 |

### 5-2. 적용 판단

Google Flights는 본 프로젝트에서 **실제 검색 기반 관측 데이터 확보 대상**으로 의미가 있다.  
따라서 분석 대상 가격은 “장부 가격”보다 **현재 검색 화면에서 소비자가 마주하는 가격 정보**로 해석한다.

## 6. 수집 방식 비교 — HTML 파싱 vs 인터셉트

### 6-1. HTML 직접 파싱

HTML 직접 파싱 방식은 화면 렌더링 결과를 그대로 해석하는 방식이다.

장점:
- 화면에 실제로 보이는 요소를 직접 읽을 수 있음
- 구현 개념이 직관적임

한계:
- 편도 + 왕복 + DPD120 + 다중 노선 조합 기준으로 비효율적
- DOM 구조 변경에 취약
- 렌더링 이후 전체 HTML 처리 비용이 큼
- 장기 운영 시 유지보수 부담이 큼

### 6-2. 인터셉트 방식

인터셉트 방식은 브라우저 자동화 과정에서 요청/응답을 가로채 원본 데이터를 확보하는 방식이다.

장점:
- HTML 본문 전체 파싱보다 효율적
- 필요한 응답 단위만 저장 가능
- raw 원본 보존과 후처리 분리가 쉬움
- 파싱 로직과 화면 로직을 분리하기 좋음

한계:
- 정확한 대상 요청/응답 식별이 필요
- 차단 및 탐지 리스크가 여전히 존재
- 응답 구조 변경 시 parser 수정 필요
- 초기 검증 난이도가 높음

### 6-3. 적용 판단

v0.4에서는 **HTML 직접 파싱을 메인 경로로 두지 않고**,  
브라우저 자동화 + 인터셉트 기반 수집을 우선 기준으로 둔다.

즉,  
**화면은 브라우저가 렌더링하고, 저장 대상은 인터셉트된 원본 응답으로 확보하는 방향**을 채택한다.

## 7. 차단 및 정책 리스크

### 7.1. 기본 인식

크롤링 기반 수집은 API 기반 수집과 달리 **차단, 탐지, 응답 구조 변경** 리스크를 내재한다.

### 7-2. 현재 기준 해석

본 프로젝트는 아래 사항을 전제로 한다.

- 차단 리스크는 해소된 문제가 아니라 **운영 리스크로 남아 있음**
- 인터셉트 방식은 효율성을 개선하지만 차단 문제를 제거하지는 않음
- 실제 운영 가능성은 첫 수집 테스트와 반복 실행 결과를 통해 확인해야 함

### 7-3. 문서 반영 원칙

따라서 v0.4 문서에서는 크롤링 구조를 **확정된 성공 경로**로 표현하지 않고,  
**기준 방향으로 채택했으나 운영 가능성은 후속 검증 대상**으로 명시한다.

## 8. 인프라 및 저장소 적용 판단

### 8-1. 수집 서버

수집 서버는 **Oracle Cloud A1 Free Tier**를 우선 기준으로 둔다.

적용 이유:
- 비용 부담 최소화
- 반복 수집 자동화 운영 가능
- 캡스톤 범위 내에서 운영 환경 경험 확보 가능

### 8-2. DB 사용

데이터 저장은 **MySQL 우선**으로 둔다.

적용 이유:
- 구조화된 관측 데이터 관리 용이
- 학습 목적상 DB 설계/적재 경험 확보 가능
- 이후 앱 조회 구조와 연결하기 쉬움

보조안:
- 구현 과정에서 필요 시 CSV / JSON fallback 허용

### 8-3. 저장 원칙

원본 인터셉트 데이터는 보존 가치가 높으므로 raw 계층에 저장한다.  
정규화된 분석/앱 조회용 데이터는 별도 테이블 또는 계층으로 분리한다.

즉 저장 방향은 아래와 같다.

> 수집 → raw 원본 저장 → 파싱 → 정규화 → MySQL 적재

## 9. 외부 데이터(Kaggle) 활용 판단

### 9-1. 후보 데이터

현재 외부 보조 후보는 `dilwong/FlightPrices`이다.

### 9-2. 활용 가능성

활용 가능 이유:
- `searchDate + flightDate` 구조로 DPD 계산 가능
- 반복 관측 패턴을 다루는 사전학습 보조로 활용 가능
- 파이프라인 및 라벨링 실험용으로 유용할 가능성 존재

### 9-3. 제한 사항

| 항목 | 내용 |
|---|---|
| 도메인 차이 | 미국 국내선 중심 |
| DPD 한계 | 1~65 범위 중심 |
| 직접 이전 불가 | 절대 가격 수준과 최종 임계값 이전 불가 |
| 역할 제한 | 메인 데이터 대체가 아니라 보조 후보 |

### 9-4. 적용 판단

Kaggle 데이터는 직접 수집 데이터가 부족한 경우의 **사전학습/패턴 보조 후보**로만 유지한다.  
최종 라벨 기준, 임계값, calibration은 반드시 직접 수집 데이터 기준으로 설정한다.

## 10. 와이어프레임 및 결과물 구체화

v0.4에서는 데이터 수집 구조와 함께 **대시보드형 앱 결과물 방향**도 구체화되었다.

현재 확인 가능한 화면 축은 아래와 같다.

- 홈 / 검색 & 요약
- 가격 분석 (BUY/WAIT)
- 요인 분석
- 알림 설정
- 환경 설정

현재 프로토타입은 Manus 기반으로 생성되었으며,  
화면 구조와 기능 배치는 v0.4 결과물 방향을 설명하는 보조 자산으로 사용한다.

단, 현재 프로토타입은 최종 앱이 아니라 **기능 방향과 화면 구성 초안**으로 해석한다.
