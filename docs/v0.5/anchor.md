## 1. 문서 목적

본 문서는 이전 문서를 참조하지 않아도 현재 프로젝트 상태를 완전히 파악할 수 있도록 작성한다.

작성 기준일: 2026-04-26

---

## 2. 프로젝트 개요

**서비스명**: AirChoice  
**팀명**: 둥근해가 떴습니다 (강주광, 이수진, 노경엽)  
**목적**: 인천(ICN) ↔ 도쿄(NRT/HND) 노선 항공권 가격 추이를 분석하여 "지금 살지(BUY), 더 기다릴지(WAIT)"를 추천하는 의사결정 지원 서비스  
**결과물 형태**: React 웹앱 (카카오 로그인, 항공권 검색 및 추천 화면)

---

## 3. 상태 구분 기준

- **확정**: 현재 운영 기준으로 채택한 항목
- **보류**: 후보로 유지하되 현재 메인 진행 대상이 아닌 항목
- **미확정**: 후속 결정 또는 구현 결과 확인 후 결정할 항목
- **폐기**: 검토했으나 현재 진행 대상에서 제외한 항목

## 4. 데이터 수집

### 4-1. 수집 방식

| 항목 | 내용 | 상태 |
|---|---|---|
| 수집 소스 | Google Flights (GetBookingResults 인터셉트) | 확정 |
| 수집 대상 | 항공사 직판 가격, 직항 편 한정 | 확정 |
| 수집 노선 | 편도 4노선: ICN→NRT, ICN→HND, NRT→ICN, HND→ICN | 확정 |
| 왕복 수집 | 편도 노선 기반 동일 항공사 출발+복귀 조합 | 확정 |
| DPD 범위 | 1~120일 (출발 120일 전부터 하루 전까지 매일 관측) | 확정 |
| 수집 주기 | 1일 3회 (00:00 / 08:00 / 16:00 KST) | 확정 |
| 수집 도구 | Python + Playwright (headless, ARM64 호환) | 확정 |

### 4-2. 수집 운영 현황

| 항목 | 내용 | 상태 |
|---|---|---|
| 운영 시작 | 2026-04-16 | 확정 |
| 운영 방식 | Oracle Cloud A1 서버, systemd timer | 확정 |
| 누적 수집 | 약 11일치 (2026-04-26 기준) | 확정 |
| 모니터링 | Discord 웹훅 (수집 성공/실패 알림) | 확정 |
| DB 백업 | Google Drive 자동 백업 (systemd timer) | 확정 |

### 4-3. 수집량 추정

| 구분 | DPD 1일당 건수 | 비고 |
|---|---|---|
| 편도 | 약 70건 | 4노선 기준 |
| 왕복 | 약 300건 | 출발+복귀 조합 기준 |

3주 수집 기준 노선 정보 충분 수준으로 판단. 외부 보조 데이터(Kaggle) 불필요 가능성 높음. 최종 판단은 수집 데이터 1차 점검 후 결정.

## 5. DB 스키마

### 5-1. 테이블 구성

| 테이블 | 역할 |
|---|---|
| search_observation | 검색 1회 부모 관측 단위 (편도/왕복 공통) |
| flight_offer_observation | 항공권 offer 행 (편도: 카드 1개 1행 / 왕복: 조합 1건 1행) |
| capture_file_log | 수집 원본 파일 경로 추적 |

### 5-2. 주요 설계 원칙

- 편도/왕복 단일 테이블 운영, `route_type` 컬럼으로 구분
- 왕복 확장 필드: `ret_` 컬럼 6개 (ALTER TABLE 서버 적용 완료, 2026-04-11)
- 편도 row의 `ret_` 컬럼은 전부 NULL
- `price_krw`: 해당 row의 상품 총가격 (편도=편도가, 왕복=왕복 총가)
- 상세 DDL: `docs/v0.4.3/schema.md` 참조

### 5-3. 미확정

- 중복 판정 키 최종안 및 UNIQUE 제약 적용 여부
- FastAPI ↔ MySQL 연결 전환 (현재 백엔드는 JSON 파일 기반 store 운영)
